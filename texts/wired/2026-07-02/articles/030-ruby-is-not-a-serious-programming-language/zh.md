# Ruby 不是一门严肃的编程语言

Section: Sheon Han
Date: 2025-12-01T07:00:00-05:00
Rubric: Ruby 依靠的是喜爱而不是实用性。我们继续往下看。

我有一个小理论：心理学中的“印刻”概念几乎可以同样应用到编程中。就像一只小雏鹅会把它看到的第一个活动生命体认作父母一样，初学程序员会对第一门成型语言中的思维模式和特质形成难以消除的依恋。

对很多人来说，那门语言就是 Ruby。它常被认为让编程“恍然开窍”；那些被它“印刻”过的人会带着某种感激与亲近来谈论它。我能理解这种感觉。我写的第一段“Hello world”是在一种叫 Java 的糟糕语言里写的，但直到学习了 JavaScript（我知道，我知道）和 OCaml，编程才开始变得直觉化，而这两门语言也根本性地塑造了我的口味。

我对 Ruby 的接触算是有点晚。直到第四份工作时，我才发现自己加入了一个主要使用 Ruby 的团队。那时我已经听够了对它优雅风格的颂赞，心中充满期待，准备被它吸引，体验那些拥护者描述的那种职业上的顿悟般感受。然而我对它的厌恶是瞬间产生的。

一个晚到达某种语言的人，会更清醒地看到它，而不带有“印刻”带来的那层宽容迷雾——也就是那种把缺陷当成性格小缺点的惯性宽恕。我看到的不是一件镶嵌着宝石的工具，而是一个可怜的小玩意儿，似乎还没收到编程世界早已前进的消息。

1995 年，Ruby 由日本程序员松本行弘（Yukihiro Matsumoto）创建，他常被昵称为“松本（Matz）”。除了创造了唯一一门起源于非西方世界的主要编程语言之外，这位出生于大阪、身为摩门教徒的程序员还以人特别友好著称，以至于 Ruby 社区采纳了座右铭 MINASWAN，即“Matz Is Nice And So We Are Nice（Matz 很友好，我们也都很友好）”。

与它好听的名字一样，Ruby 也相当“好看”。它的语法很简单，没有分号和方括号。甚至比以易读性闻名的 Python 还要简洁，Ruby 的可读性几乎像英文原文那样顺畅。

编程语言通常分为两类：静态类型和动态类型。静态类型系统像一套乐高积木，只有形状和尺寸正确的积木才能拼接到一起，从而让某些错误在物理上不可能发生。动态类型则允许你随心所欲地把积木拼在一起。从理论上看这在小规模上更灵活，但在构建大工程时这种自由会适得其反——某些错误要等到程序运行时才会被发现。换句话说，一旦你把重量压在自己的乐高独木桥上，它就会塌成一堆没用的废墟。

你大概也猜到了，Ruby 是动态类型语言。Python 和 JavaScript 也是如此，但这些社区多年来已经开发出成熟工具，让它们的行为更为“守规矩”。Ruby 目前的做法都远远达不到那个水平。它太容易犯程序员口中的“陷阱”——也就是让你非常轻易就把自己弄伤的设计。

更关键的是，Ruby 的性能表现长期稳居主流语言的底部（也就是最慢的那一档）。你可能还记得推特那张臭名昭著的“失败鲸鱼”错误页——服务宕机时出现、由小鸟托举着一只鲸鱼的那张图。可以说，Ruby 在其中负有相当大的责任。2010 年世界杯期间推特的崩溃让人们警醒，随后公司决定将后端迁移到 Scala，这是一种更稳健的语言。

这次迁移是值得的：到 2014 年世界杯时，推特在决赛场次处理了 3200 万条推文，且没有发生中断。它基于 Scala 的新后端处理能力比 Ruby 快了最多 100 倍。在 2010 年代，陆续有一批公司替换了大部分 Ruby 基础设施，而当旧有的 Ruby 代码仍需保留时，新的服务则改用性能更高的语言编写。

更进一步说，如今 Ruby 能做的事，几乎都有其他语言做得更好，这使它失去了鲜明的专属领域。对于快速脚本和自动化，Python、JavaScript 和 Perl 都是强有力的竞争对手。Python 尽管也不算快，却在科学计算中开辟了主导地位，并成为事实上的人工智能语言。JavaScript 则成功统治了网页端。至于 Perl，嗯，它正在走向衰退——这点让我不觉得可惜。Ruby 现在处在一个很尴尬的中间地带。

你可能会好奇，为什么到 2025 年人们还在使用 Ruby。它能存活下来，是因为与 Ruby on Rails 的共生关系——正是这个 Web 框架促成了 Ruby 的广泛采纳，并持续支撑着它的现实关联性。

当丹麦开发者大卫·海涅迈耶·汉森（DHH）在 2004 年发布 Rails 时，Ruby 不再只是一些“可爱日本程序员”的专属领地。DHH 与 Matz 颇像一张底片底色——同样是张扬的人物，魅力与教条各占一半，所谓 MINASWAN 也是如此，而他又乐于在公开的 Twitter 辩论中胜出，并在业余时间赢得赛车锦标赛。

在 Web 2.0 崛起与 Facebook 上市之间的那个朴素年代，当硅谷正沉浸在 TechCrunch Disrupt 的喜悦之中，并且还完全不知道即将到来的科技反噬时，Rails 是新一代创业公司首选的框架。Airbnb、GitHub、Twitter、Shopify 和 Stripe 的核心代码库都是基于它构建的。

在 2000 年代初期，构建 Web 应用非常繁琐，Rails 为开发者提供了一个一站式方案。它不必再让开发者手动把数据库、前端和后端一个个拼接起来，而是提供了一个直接可用的成套解决方案。那是一种“完整整合”，就像弗兰克·劳埃德·赖特设计的一座紧凑乌索尼亚住宅，每个细节都围绕着单一统一的设计理念精心安排。

然而，那也是一个低估了互联网终将变得多么庞大的时代。就像你试图改造一套乌索尼亚住宅的厨房或加建第二层时，统一性反而会变成负担。最初只为容纳少量规整有礼的晚宴客人而设计的房子，如今却被要求更像一个可容纳百万名难以管束访客的会议中心。

并非只有我一个人对 Ruby 持悲观看法。在 Stack Overflow 的年度开发者调查中，它多年来的人气一直在下降：从 2013 年的前十名技术，一直滑落到今年的第 18 位，甚至还落后于汇编语言。在新一代开发者中，Python 和 JavaScript 的排名要高得多。Ruby 现在暂时仍然存在，像一种职业安心对象一样，靠的是遗留代码库的惯性以及那些最早接触并深爱它的人提供的忠诚支撑。但怀旧情绪和一个好听的名字，终究撑不起一切。
